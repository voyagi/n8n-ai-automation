**Rewire connections (replace existing Route by Spam Score connections):**

Current connections from "Route by Spam Score":
- output 0 (spam) → Spam Response
- output 1 (legitimate) → Success Response

New connections:
- "Route by Spam Score" output 0 (spam) → "Flag as Spam"
- "Route by Spam Score" output 1 (legitimate) → "Flag as Legitimate"
- "Flag as Spam" output → "Log to Google Sheets" AND "Spam Response" (both in same output array)
- "Flag as Legitimate" output → "Log to Google Sheets" AND "Success Response" (both in same output array)

Wait — the Sheets node should run BEFORE the response nodes (storage first, then respond). The connection pattern should be:

- "Route by Spam Score" output 0 → "Flag as Spam"
- "Route by Spam Score" output 1 → "Flag as Legitimate"
- "Flag as Spam" output → "Log to Google Sheets" (index 0)
- "Flag as Legitimate" output → "Log to Google Sheets" (index 0)
- "Log to Google Sheets" output → BOTH "Spam Response" AND "Success Response"

BUT this creates a problem: after the Sheets node, how does the workflow know which response node to use? The Sheets node receives items from both branches but outputs them identically. We lose the branch context.

**Correct approach (per research Pattern 1 — dual branch convergence):**
Instead of converging BEFORE response, do BOTH in parallel from the flag nodes:

- "Flag as Spam" → ["Log to Google Sheets", "Spam Response"] (parallel outputs)
- "Flag as Legitimate" → ["Log to Google Sheets", "Success Response"] (parallel outputs)

This way:
1. Both branches write to Sheets (convergence for storage)
2. Each branch sends its own HTTP response (branch-specific)
3. n8n executes the Sheets node once per active branch
4. No need to re-split after convergence

**Connection JSON structure:**
